# 소싱·정규화 가이드 (SOURCING.md)

> v3.0 — 2026-07-14. **SSOT: 이 파일 (chaekbi-sources 레포).**
> v2.0(웹 프로젝트 상주)에서 이전. 웹 "책비-소스레포" 프로젝트는 소싱 **판단** 협의처이며, 규칙 원본은 항상 이 파일이다. 웹 사본과 충돌 시 이 파일 우선.
> v3.0 변경: ① SSOT를 레포로 이전 ② 단편형(원문 서고) 파일 규칙 정식 편입 — korean-modern-anthology 실무 표준을 축복 ③ 소싱·정규화·커밋 실행 주체를 CC로 명시.
> 정규화 형식의 사실상 표준은 frankenstein/ 커밋본이며, 이 문서는 그것을 명문화한 것.
> **작품 고유 결정(저본·판본·편 선정·마커)은 전부 해당 작품의 SOURCE.md에. 작품 목록의 원본은 README.md에.** 이 문서는 유형별 규칙만 담는다 — 작품이 늘어도 이 문서는 안 늘어난다.

## 0. 역할 분담

- **웹 허브/소스레포 프로젝트**: PD 판별 애매건, 저본·이본 결정, 소스 유형 신설 등 **판단**이 필요한 협의. 결정 결과는 이 문서 또는 해당 SOURCE.md에 반영해 커밋한다 — 대화에만 남은 결정은 결정이 아니다.
- **CC (Claude Code)**: fetch·정규화·마커 검증·SOURCE.md 작성·커밋 — **실행 전량**. 원어 소스 사이트 접근이 필요하면 CC 환경 allowlist에 추가한다(웹 환경은 zh/ja 위키문헌·아오조라 등 차단).
- **성경**: 저본·유형·수록 범위의 최종 결정. CC가 수동 확보 불가한 소스는 성경이 확보해 전달.

## 1. 퍼블릭 도메인 판별 (모든 유형 공통 — 커밋 전 필수)

- **원작**: 저자 사후 70년 경과 (한국법 기준). 단, 1962년 이전 사망 저자는 구법 사후 50년 규정으로 이미 만료 확정(2013년 개정법 소급 적용 없음).
- ❗**진짜 함정은 원작이 아니라 그 위에 얹힌 저작권이다.** 커밋하려는 **바로 그 텍스트**의 권리를 따진다:
  - 번역본 → 번역자 저작권 별도
  - 현대어 교주본·주석본 → 편집저작권
  - 위키문헌 현대역 → CC BY-SA 라이선스 조건
  - 즉 "원작이 PD"는 근거가 못 된다. **커밋 파일의 텍스트가 PD인 근거**를 SOURCE.md에 적을 수 있어야 커밋 가능.
- 공개 repo이므로 애매하면 커밋하지 않는다. 판별부터 — 판별 협의는 웹에서.

## 2. 소스 유형 분류 (착수 시 가장 먼저 판별)

새 작품 착수 시 아래 중 하나로 분류하고, 유형을 SOURCE.md에 명기한다. 어느 유형에도 안 맞으면 가이드를 고치기 전에 웹 허브와 유형 신설 여부부터 논의.

### A형 — 영어 원문 (함정 없음)
- 예: 프랑켄슈타인, 오만과 편견, 드라큘라, 지킬 앤 하이드, 셜록 홈즈
- 저본 = 영어 원문 그대로. 소싱: GITenberg 미러 우선, gutenberg.org 캐시 예비.
- 유일 주의점: **판본 계열** (구텐베르크에 같은 작품 복수 eBook 존재) — 어느 번호·어느 판본 계열인지 확정하고 마커로 검증.

### B형 — 비영어 서양 고전 (번역 저본)
- 예: 몬테크리스토 백작(프랑스어), 레미제라블(프랑스어)
- 원어 원문을 저본으로 쓰기 어려우면 **PD가 확인된 번역본을 저본**으로 한다.
- SOURCE.md 필수 기록: 번역자(익명이면 익명 표기)·출간연도·PD 근거. "번역본이 저본"임을 명기.
- 번역본이 여럿이면 저본 하나로 고정 — 판본 섞기 금지 (A형의 판본 고정 원칙과 동일).

### C형 — 한문·고전어 동아시아 고전 (원전 직행)
- 예: 요재지이(한문), 삼국유사(한문), 우게츠 이야기(고전 일본어)
- **원전 텍스트만 커밋.** 현대어역·교주본은 라이선스 불문 커밋 금지. (아오조라 문고류도 수록본이 현대어역이면 제외)
- PD 번역은 각색 시 **대조 참고용으로만** — repo에는 원칙적으로 커밋하지 않는다. 커밋이 꼭 필요하면 별도 하위 폴더 + PD 근거 명기.
- **선별 수록 허용**: 옴니버스 대작(수백 편)은 각색 대상 편만 수록. 전량 커밋 불필요. 선정 목록과 기준은 SOURCE.md에.

### D형 — 한국 고전 (이본 문제)
- 예: 홍길동전, 전우치전, 박씨전, (향후) 춘향전류
- ❗**저본(이본) 결정이 원문 확보보다 선행한다.** 결정 전 커밋 금지. 이본 명칭·소장처·선정 근거를 SOURCE.md에 필수 기록.
- 소싱: 위키문헌 **원문(옛한글 계열)만.** 현대역 페이지(CC BY-SA) 참조·커밋 금지. 대조용 아카이브(공유마당, 국립중앙도서관/장서각)는 커밋 없이 비고에 대조 사실만.

### E형 — 한국 근현대 원문 (그대로 PD)
- 예: 김유정 단편집, 원문 서고(근대 단편선)
- 원문이 이미 현대 한국어 계열 — 번역 함정 없음. 사후 70년(또는 §1의 구법 50년 만료)만 확인.
- 주의: 위키문헌 페이지가 **원전 표기의 단순 전산화인지, 현대 맞춤법 교정본인지** 확인하고 어느 쪽인지 SOURCE.md에 기록. 편집 각주·해설은 제외.
- 위키텍스트 템플릿 처리(SIC·드러냄표·복원 등)는 `action=parse` 렌더링 결과와 대조해 "화면에 표시되는" 파라미터만 남긴다 — 세부 규칙의 선례는 kim-yujeong-anthology/SOURCE.md.

### F형 — 동아시아 백화(白話) 장편소설 (원전 완결 수록)
- 예: 봉신연의, 수호전, (향후) 삼국지연의·서유기류
- 언어: 문언이 아닌 근세 백화 구어체 장회소설. C형과 달리 번역/현대역 트랩은 없음 — 원작 자체가 이미 백화문.
- **PD 함정**: 유통되는 텍스트 대부분이 현대 표점본(구두점·교감 추가)이다. 표점 자체가 편집저작물로 볼 여지는 낮으나, **SOURCE.md에 "표점이 창작적 편집이 아니라는 근거"를 명시해야 커밋 가능** — 근거 없이 표점본을 그대로 가져오지 않는다.
- **구조**: C형의 옴니버스 "선별 수록" 규정은 적용하지 않는다. 완결 장편이므로 A형처럼 **전편 수록** — 회차 번호를 그대로 살려 `ch-XX.txt`(또는 원문 표기가 "回"/"卷"이면 그에 맞는 접두어).
- **이본 문제**: D형처럼 판본 계열 확정이 원문 확보에 선행한다. 명대 소설은 판각본·평점본이 여럿 존재할 수 있음 — 저본 선정 근거를 SOURCE.md에 필수 기록.

> 소스 유형(A~F)은 **PD 함정·저본 축**의 분류다. 파일 저장 방식은 §3의 **파일 단위 축**(연재형/옴니버스형/단편형)이 별도로 정한다 — 두 축은 독립이다. (예: korean-modern-anthology는 소스 유형 E형 + 파일 단위 단편형)

## 3. 정규화 표준 (전 유형 공통 — 프랑켄슈타인 형식 = 표준)

- **원문 텍스트 무수정.** 문장·철자·구두점·문단 배치 변경 없음. 하드랩(수동 줄바꿈)도 그대로 — 제거하는 건 `\r`뿐.
- CRLF → LF (`tr -d '\r'`). 인코딩 UTF-8, BOM 없음.
- 소스 사이트 보일러플레이트(라이선스 머리말/꼬리말, 위키 각주 틀·편집 표기) 제외, 본문만.
- 파일 첫 줄은 챕터/편 표제. 원문에 있으면 그대로, 없으면 넣지 않는다 (임의 창작 금지).
- **파일 단위 (3종):**
  - **연재형** (장 단위로 순서가 이어지는 소설): `ch-XX.txt` (zero-pad, 자릿수는 총 화수에 맞춤). 본문 외 구조는 성격에 맞는 접두어 (예: `letter-XX.txt`, `preface.txt`, `epilogue.txt`).
  - **옴니버스형** (독립된 편/이야기의 모음, 각색 파이프라인 대상): 편별 개별 파일 `tale-XXX.txt` 또는 원문 단위에 맞는 접두어(`juan-XX.txt`, `gwon-XX.txt`). 편 순서·목차·명명 규칙은 SOURCE.md에.
  - **단편형** (독립 완결 단편 원문을 그대로 싣는 "원문 서고" — **각색 파이프라인 대상 아님**): 1파일 = 1작품, 챕터 분할 없음. 파일명 `NN-저자-제목.md`. **각 파일 선두에 YAML 프론트매터** (`title`/`author`/`year`/`type`/`source`/`source_name`/`license`)로 편별 출처를 자체 기록 — 편마다 저자·출처가 달라 SOURCE.md 한 곳에 몰 수 없기 때문. 서고 전체의 공통 사항(판본 계열·PD 근거·수록 목록)은 폴더 SOURCE.md에.
- **`.txt` vs `.md`**: **각색 파이프라인에 들어가는 원문 본문은 항상 `.txt`** — 마크다운 렌더링·서식을 원천 차단해 "원문 무수정" 원칙을 파일 형식 차원에서 강제하기 위함. **`.md` 원문은 단편형(원문 서고)에서만 허용** — 프론트매터가 필요하고, 파이프라인이 아닌 앱이 직접 소비하는 형식이기 때문. 이 외의 `.md`는 `SOURCE.md`·`README.md` 등 메타데이터 문서 전용.

## 4. SOURCE.md (작품당 1개 — 작품 고유 사항의 유일 기록처)

프랑켄슈타인 SOURCE.md가 서식 원본. 필수 항목:

- 소스 유형 (§2 A~F) + 파일 단위 (§3 연재형/옴니버스형/단편형)
- 원저자 / 원제 / 판본(저본) + **저본 선정 근거**
- 소스 경로: 표준 + 예비
- **판본 검증 마커**: 해당 판본에만 있는 구문 2개 이상. 하드랩 때문에 구문이 줄에 걸려 갈라질 수 있으므로 검증 시 `tr -d '\r' | tr '\n' ' '`로 펴서 grep.
- 저작권 상태: **커밋 텍스트 기준** (B형은 번역자·연도·PD 근거, F형은 표점 근거까지)
- 정규화 내역 / 파일 단위 상세 (옴니버스·단편형이면 목차) / 비고
- **`[미정]` 항목이 남은 SOURCE.md는 원문 커밋 불가.**
- 컬렉션 작품(예: sherlock-holmes-casebook)은 하위 폴더별 SOURCE.md 허용 — 단위는 "저본이 하나로 고정되는 범위".

## 5. 원격 접근 메모 (웹 허브 환경용 — CC는 로컬 clone이므로 해당 없음)

- 웹 bash 환경 접근 가능: raw.githubusercontent.com, github.com, api.github.com, codeload.github.com, ko.wikisource.org.
- 그 외(zh/ja 위키문헌, 아오조라 등) 차단 — CC 환경 allowlist 추가 또는 성경이 수동 확보.
- 구텐베르크 직접 fetch는 cold fetch 거부될 수 있음 → 웹 검색 1회 후 결과 링크로 fetch, 또는 GITenberg 미러 curl.

### GitHub 저장소 원격 접근 — 용도별 방법 (검증 완료)

- **전체 repo 상태 점검**: `codeload.github.com/banbanworks-studio/chaekbi-sources/tar.gz/refs/heads/main` tarball 통째 다운로드 → `tar xzf` 후 `find`/`grep`. 가장 빠르고 확실하며 rate limit 없음. 상태 점검은 원칙적으로 이 방법만.
- **개별 파일 1개 확인**: `raw.githubusercontent.com/banbanworks-studio/chaekbi-sources/main/{경로}`.
- **파일 목록 1회성 조회**: `api.github.com/repos/.../git/trees/main?recursive=1` — 비인증 시간당 60건 제한. 반복 조회 예상 시 처음부터 tarball.
- **github.com의 `/tree/`, `/commits/` HTML 페이지는 쓰지 않는다** — robots 차단. 저장소 루트 HTML은 캐시된 부분 목록만 반환해 신뢰 불가.
- 커밋 스코프 확인은 `api.github.com/repos/.../commits/{sha}`.
